# Спецификация Event Model и SLO/SLA для WB Parser

## 1. Цель и область действия

Документ фиксирует:

- каноническую модель событий для детекции и доставки уведомлений;
- целевые значения SLO/SLA по задержке и надежности уведомлений;
- acceptance-критерии для реализации и тестов.

Область: пайплайн `parser -> event_detector -> Redis Streams -> notifier -> Telegram`.

## 2. Каноническая модель событий

### 2.1 Типы событий

Канонические типы событий (должны использоваться в интеграционных контрактах и тестах):

- `new_product_in_range`
- `entered_range`
- `price_drop`

Маппинг совместимости с текущей реализацией:

- `new_product` -> `new_product_in_range`
- `enter_range` -> `entered_range`
- `price_drop` -> `price_drop`

### 2.2 Обязательный payload события

Каждое событие ДОЛЖНО включать:

- `event_type` (одно из канонических значений)
- `task_id`
- `user_id`
- `product_id`
- `name`
- `price` (текущая наблюдаемая цена)
- `price_old` (nullable)
- `price_new` (равно `price`)
- `url` (nullable)
- `seller` (nullable)
- `detected_at_ms` (unix epoch в миллисекундах)

Пайплайн доставки ДОЛЖЕН по возможности добавлять:

- `enqueued_at_ms` (момент записи в Redis Stream)
- `notified_at_ms` (момент успешной отправки в Telegram)

### 2.3 Правила детекции

Обозначения:

- `current_price` — текущая наблюдаемая цена;
- `last_price` — предыдущая известная цена для `product_id`;
- `price_min`, `price_max` — фильтры задачи;
- `min_price_change` — порог из конфигурации.

#### `new_product_in_range`

Событие эмитится только если:

- `last_price is None`
- `current_price` в диапазоне:
  - `(price_min is None or current_price >= price_min)`
  - `(price_max is None or current_price <= price_max)`

#### `entered_range`

Событие эмитится только если:

- `last_price is not None`
- `price_max is not None`
- `last_price > price_max`
- `current_price <= price_max`

#### `price_drop`

Событие эмитится только если:

- `last_price is not None`
- `current_price < last_price`
- `abs(last_price - current_price) >= min_price_change`
- товар уже был известен для данного task/user (не первое обнаружение)

Для текущей версии `price_min` НЕ ограничивает `price_drop`; для входа в диапазон используется только явное условие пересечения границы диапазона.

## 3. Дедупликация, cooldown и политика повторов

### 3.1 Дедупликация на уровне событий

- Dedup-ключ: `event:{user_id}:{product_id}`
- Dedup-окно (`EVENT_DEDUP_TTL`): по умолчанию 300 сек.
- В пределах dedup-окна второе событие для той же пары `(user_id, product_id)` не должно эмититься.

### 3.2 Дедупликация на уровне уведомлений

- Dedup-ключ: `notify:{user_id}:{product_id}:{price}`
- Dedup-окно (`NOTIFICATION_DEDUP_TTL`): по умолчанию 12 ч.
- В пределах окна повторное уведомление для той же тройки `(user_id, product_id, price)` подавляется.

### 3.3 Модель повторных уведомлений

Уведомление считается дублем, если существует dedup-ключ уведомления.

Повтор разрешен, если выполняется хотя бы одно условие:

- истек TTL dedup-ключа;
- изменилась цена (формируется новый dedup-ключ);
- событие относится к новой детекции после истечения cooldown на уровне событий.

## 4. Семантика доставки

Для текущей production-цели:

- очередь и notifier гарантируют `at-least-once` обработку от Redis Stream до обработчика notifier;
- защита от видимых пользователю дублей best-effort на базе dedup-ключей выше;
- доставка считается успешной только после успешного ответа Telegram API.

## 5. SLO/SLA

### 5.1 Основной пользовательский SLI

`time_to_notify_ms = notified_at_ms - detected_at_ms`

Измеряется по успешно доставленным уведомлениям.

### 5.2 Целевые SLO

SLO на скользящем окне 30 дней:

- p50 `time_to_notify_ms` <= 5,000 ms
- p95 `time_to_notify_ms` <= 30,000 ms
- p99 `time_to_notify_ms` <= 90,000 ms
- доля успешной доставки >= 99.0%
- queue lag (возраст самого старого pending-сообщения) p95 <= 15,000 ms

### 5.3 SLA (базовый внешний уровень обязательств)

Месячный SLA baseline:

- 99.0% уведомлений доставляются не позже 120,000 ms от момента детекции;
- месячный error budget: 1.0% на поздние/неуспешные доставки.

Нарушения SLA расходуют error budget в случаях:

- неуспешная доставка после исчерпания ретраев;
- успешная, но поздняя доставка (выше SLA-порога).

## 6. Требования к наблюдаемости

Система ДОЛЖНА публиковать минимум следующие метрики:

- гистограмма `time_to_notify_ms` с p50/p95/p99
- gauge queue lag (`queue_lag_ms`)
- счетчики успехов/ошибок доставки
- счетчики событий по типам
- подавления дедупликацией по слоям (`event_dedup`, `notification_dedup`)

Каждый поток уведомления ДОЛЖЕН трассироваться по `(task_id, user_id, product_id, event_type)`.

## 7. Acceptance-критерии

Спецификация считается принятой, когда выполнены все критерии ниже.

### 7.1 Критерии по Event Model

- Detector эмитит только канонические типы событий (или канонизированные алиасы на границе).
- Для каждого типа события есть позитивные и негативные unit-тесты условий.
- Event payload содержит все обязательные поля.

### 7.2 Критерии по dedup/cooldown

- Дедуп событий подавляет повторную эмиссию в пределах `EVENT_DEDUP_TTL`.
- Дедуп уведомлений подавляет ту же тройку `(user, product, price)` в пределах `NOTIFICATION_DEDUP_TTL`.
- Повторное уведомление разрешается после истечения TTL или при изменении цены.

### 7.3 Критерии по задержке/SLO

- Integration-тест подтверждает, что `detected_at_ms` и `notified_at_ms` заполняются и пригодны для расчета.
- Нагрузочный сценарий (>=100 активных задач) публикует отчет с p50/p95/p99 `time_to_notify_ms`.
- Результаты удовлетворяют SLO или помечаются как release blocker.

### 7.4 Критерии по надежности

- End-to-end integration-тест проходит: `parser -> Redis Stream -> notifier -> telegram mock`.
- Семантика at-least-once подтверждается при рестарте воркера (без потери сообщений).
- Тест на отказоустойчивость подтверждает ретраи и корректный учет ошибок в метриках.

### 7.5 Release Gate

Выход в production блокируется, если:

- не пройдены все acceptance-критерии выше в CI/staging;
- не включены SLO-дашборд и алерты по p95/p99 latency, queue lag и delivery failures.
